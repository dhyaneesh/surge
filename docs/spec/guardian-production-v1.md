# Guardian SRE

## Production-Ready V1 Project Specification

## 1. Executive summary

Guardian SRE is an environment-agnostic action and verification layer for SigNoz and Kubernetes.

It combines three operating modules:

1. **SRE Sidekick** investigates incidents and determines what is failing and why.
2. **Deploy Guardian** determines whether a release caused a regression and coordinates a policy-controlled rollback.
3. **Guardian Scale** determines whether legitimate demand requires scaling and exposes approved SigNoz demand signals to KEDA.

The product must observe an incident, determine whether the appropriate response is investigation, scaling, rollback, or alerting, execute only an authorized action, and verify recovery using fresh telemetry.

The originating product concept explicitly combines incident investigation, deployment correlation, scaling, policy decisions, typed action providers, and post-action SigNoz verification.
The central product capability is not merely detecting an error or latency increase. Guardian must distinguish between situations that produce similar symptoms but require different actions:

```text
Legitimate load increase
→ scale

Newly deployed version regressed
→ rollback

Caller latency increased because a dependency is saturated
→ do not blindly scale the caller

Telemetry or evidence is insufficient
→ investigate or alert
```

Guardian must be production-ready for on-premises Kubernetes and remain general enough to support applications beyond the reference demo environments used during development.

# 2. Product goals

## 2.1 Primary goals

Guardian must:

* Ingest alerts, deployment events, Kubernetes changes, and operator questions.
* Retrieve metrics, traces, logs, exceptions, deployment markers, and infrastructure state.
* Produce typed incident assessments.
* Rank competing hypotheses.
* Record supporting and contradicting evidence.
* Distinguish legitimate demand from software regressions and dependency failures.
* Recommend scaling policies from historical telemetry.
* Expose deterministic SigNoz metrics to KEDA.
* Propose and execute policy-controlled rollback actions.
* Require approval for destructive production actions by default.
* Verify whether remediation restored service health.
* Preserve a complete audit and replay history.
* Support air-gapped or restricted-egress on-premises installations.
* Work across different application architectures, languages, protocols, and deployment systems.

## 2.2 Non-goals

Production V1 will not:

* Allow an LLM to directly patch Kubernetes resources.
* Put an LLM inside the KEDA polling loop.
* Automatically roll back production workloads by default.
* Generate arbitrary runtime queries for the external scaler.
* Replace SigNoz dashboards or alerting.
* Become a general unrestricted Kubernetes operations agent.
* Perform arbitrary shell commands generated by a model.
* Require any one application architecture, protocol, database, queue, or deployment system.
* Require the OpenTelemetry Demo or any other demo application to function.

# 3. Product modules

## 3.1 SRE Sidekick

SRE Sidekick is Guardian’s shared incident-investigation layer.

It handles:

* SigNoz alerts.
* Slack and UI investigation requests.
* Metrics, traces, logs, and exceptions.
* Baseline comparisons.
* Deployment correlation.
* Traffic and load analysis.
* Dependency analysis.
* Resource saturation.
* Competing hypotheses.
* Contradicting evidence.
* Telemetry-quality validation.
* Action recommendation.
* Recovery explanation.

Its output must be a typed assessment:

```python
class IncidentAssessment(BaseModel):
    tenant_id: UUID
    incident_id: UUID
    scoring_version: str
    evidence_snapshot_id: UUID

    incident_class: Literal[
        "load_spike",
        "deployment_regression",
        "dependency_failure",
        "resource_saturation",
        "telemetry_failure",
        "unknown",
    ]

    affected_services: list[ServiceRef]
    hypotheses: list[Hypothesis]
    evidence_ids: list[UUID]
    contradicting_evidence_ids: list[UUID]
    missing_evidence: list[str]

    telemetry_quality: float
    evidence_confidence: float
    model_confidence: float
    deterministic_winner_hypothesis_id: UUID | None
    model_tie_break_used: bool
    model_tie_break_reason: str | None
    conflict_detected: bool

    proposed_action: (
        ScaleAction
        | RollbackAction
        | AlertOnlyAction
        | InvestigationAction
        | PauseScalerAction
        | None
    )
```

The Sidekick must not always recommend the same remedy. The decision between scale, rollback, investigation, and no action is the product’s central intelligence.

## 3.2 Deploy Guardian

Deploy Guardian ingests CI/CD changes and determines whether a recent release caused a measurable regression.

It must:

* Ingest deployment events.
* Map deployments to observed service versions.
* Compare pre-deployment and post-deployment telemetry.
* Compare stable and newly deployed versions.
* Identify newly appearing error signatures.
* Rank failing operations.
* Inspect representative traces.
* Check dependency changes.
* Rule out traffic and infrastructure changes.
* Retrieve relevant commit, pull request, configuration, and manifest differences.
* Propose rollback or alerting.
* Execute through a typed rollback provider.
* Verify recovery with fresh SigNoz evidence.

The source product concept requires this sequence of version comparison, error analysis, representative trace inspection, dependency checks, change retrieval, and rollback or alert proposal.

## 3.3 Guardian Scale

Guardian Scale has two subcomponents.

### Autoscaling Advisor

The advisor analyzes historical SigNoz data and recommends:

* Scaling metric.
* Query scope.
* Target value per replica.
* Minimum replicas.
* Maximum replicas.
* Scale-up velocity.
* Scale-down stabilization.
* Cooldown period.
* Blocking dependencies.
* Cost or capacity limits.
* Shadow-mode validation requirements.

The advisor may produce a reviewable `GuardianScalingPolicy` and KEDA `ScaledObject`.

The advisor does not participate in every scaling poll.

### Guardian KEDA External Scaler

The external scaler is a deterministic gRPC service.

It must:

* Implement KEDA’s external-scaler protocol.
* Load an approved scaling policy.
* Verify the policy’s immutable content hash.
* Execute an approved Query Contract.
* Validate freshness, units, sample counts, and result shape.
* Return only numeric metric values.
* Fail safely when SigNoz is unavailable.
* Never call an LLM.

The original architecture explicitly keeps the LLM outside the KEDA polling loop and requires deterministic policy and query execution.

# 4. Architectural principles

## 4.1 Environment-agnostic core

Guardian must reason using normalized product concepts:

* `ClusterRef`
* `EnvironmentRef`
* `ServiceRef`
* `WorkloadRef`
* `DeploymentRef`
* `DependencyRef`
* `TelemetrySignal`
* `QueryContract`
* `ActionProposal`
* `RecoveryContract`

Guardian core logic must not contain application-specific concepts such as:

* Astronomy Shop.
* Online Boutique.
* Checkout service.
* RabbitMQ.
* Redis.
* Argo Rollouts.
* AWS Retail Store.

Those names may exist in configuration, evidence, tests, or provider adapters, but must not define product behavior.

A new Kubernetes application must be onboarded through configuration and provider integration rather than modifications to Guardian’s core reasoning engine.

## 4.2 Deterministic execution, probabilistic analysis

Deterministic evidence scoring is authoritative.

The LLM may:

* Generate candidate hypotheses.
* Select allowlisted evidence tools.
* Interpret source and configuration changes.
* Explain evidence.
* Produce structured assessment output.
* Suggest scaling-policy parameters.
* Break a near-tie only when every tied hypothesis has already passed deterministic eligibility gates.

The LLM may not:

* Change an evidence score or hypothesis score.
* Promote an ineligible hypothesis.
* Reject a deterministically dominant eligible hypothesis.
* Authorize actions.
* Calculate live replica counts.
* Return KEDA metric values.
* Execute Kubernetes writes.
* Approve rollback.
* Determine whether an approval is valid.
* Evaluate final recovery conditions.
* Generate arbitrary runtime SQL, PromQL, ClickHouse SQL, or backend query fragments for an active scaler.

Every assessment must record the deterministic scoring version, eligibility result, tie-break eligibility, model and prompt version, and final reason code.

## 4.3 Evidence before action

Every action proposal must reference immutable evidence records.

An action must be rejected when:

* Evidence is missing.
* Evidence is stale.
* Telemetry quality is insufficient.
* The target environment is ambiguous.
* Service and workload identity cannot be resolved.
* Deployment identity cannot be mapped to a running version.
* The rollback target no longer exists.
* A newer deployment superseded the candidate.
* Contradicting evidence exceeds the permitted threshold.
* A conflicting action is already active.

## 4.4 Durable workflows

Incidents may remain active while Guardian:

* Waits for telemetry windows.
* Retries evidence collection.
* Waits for approval.
* Monitors rollout progress.
* Verifies multiple recovery windows.
* Escalates failed recovery.

These workflows must use Temporal. Database flags, cron jobs, NATS consumers, or Kubernetes controllers must not replace the Temporal state machine.

Each incident owns exactly one `IncidentWorkflow` parent. Deployment assessment, action execution, and recovery verification run as child workflows with deterministic child workflow IDs. Duplicate event delivery may signal the existing parent but must never create a second parent for the same tenant and incident.

## 4.5 Policy independent of reasoning

The model proposes an action.

OPA decides whether that action is permitted.

OPA evaluates:

* Environment.
* Incident severity.
* Evidence confidence.
* Telemetry quality.
* Proposed parameters.
* Target workload.
* Replica bounds.
* Approval identity.
* Change freezes.
* Separation-of-duties requirements.
* Proposal expiry.
* Concurrent action state.
* Rollback-target availability.

## 4.6 Fresh verification after every action

Evidence used to authorize an action must not be reused to verify success.

Recovery must use fresh SigNoz queries collected after:

* Workload convergence.
* An observation delay.
* The configured evaluation window.

## 4.7 Normative architecture requirements and default thresholds

The requirements in this section are normative for Production V1. A deployment may override a numeric default only through a versioned configuration change that passes the same review, audit, and rollout controls as policy. Overrides must not weaken a fail-closed invariant.

### 4.7.1 Deterministic hypothesis authority

Requirements:

* `GRD-REA-001`: Guardian must calculate evidence and hypothesis scores using a versioned deterministic implementation.
* `GRD-REA-002`: A hypothesis is eligible only when `deterministic_score >= 0.70`, telemetry is healthy, required identity is resolved, and normalized contradiction weight is `<= 0.25`.
* `GRD-REA-003`: A mutating action may be proposed only when `evidence_confidence >= 0.85`.
* `GRD-REA-004`: The LLM may break a tie only when all tied hypotheses are eligible and the score difference between the highest and lowest tied hypothesis is `<= 0.03`.
* `GRD-REA-005`: The LLM tie-break may select among eligible hypotheses but must not modify scores, evidence links, eligibility, or action gates.
* `GRD-REA-006`: When the highest eligible hypothesis leads the next eligible hypothesis by more than `0.03`, the deterministic ordering is final.
* `GRD-REA-007`: Evidence sharing an `independence_group` must contribute at most once to independent support and contradiction totals.

State transitions:

```text
CANDIDATE
→ DETERMINISTICALLY_REJECTED
→ DETERMINISTICALLY_ELIGIBLE
    → DETERMINISTIC_WINNER
    → NEAR_TIE_ELIGIBLE
        → MODEL_TIE_BREAK_SELECTED
        → TIE_BREAK_UNAVAILABLE
```

Failure behavior:

* Invalid model output, model timeout, unavailable inference, or disagreement with deterministic eligibility must not select an action.
* `TIE_BREAK_UNAVAILABLE` transitions to conflict resolution or continued investigation.
* A scoring implementation or configuration version mismatch blocks mutating action proposals.

### 4.7.2 `telemetry_failure` and `unknown`

Requirements:

* `GRD-CLS-001`: Telemetry is healthy only when `telemetry_quality >= 0.80` and no critical integrity check fails.
* `GRD-CLS-002`: A critical telemetry integrity failure is any of:
  * environment, tenant, service, or workload identity is missing or conflicting;
  * the newest required sample is older than `2 × QueryContract.freshness_seconds`;
  * timestamp skew exceeds `60 seconds`;
  * a required signal has zero usable samples;
  * the relevant collector or ingestion pipeline is known to be unavailable;
  * cardinality collapse or sampling change invalidates comparison.
* `GRD-CLS-003`: Any critical telemetry integrity failure produces incident class `telemetry_failure`, regardless of causal hypotheses.
* `GRD-CLS-004`: Incident class `unknown` is permitted only when telemetry is healthy and no causal hypothesis remains eligible after the configured evidence-collection budget.
* `GRD-CLS-005`: `telemetry_failure` may propose only investigation, alerting, or scaler pause. It must not propose rollback, scale-down, or version-specific mutation.
* `GRD-CLS-006`: The default evidence-collection budget is two collection passes and `10 minutes` of wall-clock time, whichever is reached first.

State transitions:

```text
TELEMETRY_VALIDATION
→ TELEMETRY_HEALTHY
    → CAUSAL_ASSESSMENT
        → CLASSIFIED
        → UNKNOWN
→ TELEMETRY_UNHEALTHY
    → TELEMETRY_FAILURE
```

Failure behavior:

* Guardian must not convert an unhealthy-telemetry result to `unknown`.
* A telemetry-quality evaluator failure is itself treated as `telemetry_failure`.
* Recovery verification cannot succeed while telemetry is unhealthy.

### 4.7.3 Conflicting action recommendations

Requirements:

* `GRD-ACT-001`: Simultaneously eligible hypotheses conflict when they imply mutually exclusive actions, including scale versus rollback, scale versus dependency protection, rollback versus continue investigation, or any pair whose action domains overlap incompatibly.
* `GRD-ACT-002`: A conflict must transition the incident to `CONFLICT_RESOLUTION`; Guardian must not choose one mutating action.
* `GRD-ACT-003`: Conflict-resolution evidence requests must be selected for discriminatory value between the conflicting hypotheses.
* `GRD-ACT-004`: Conflict resolution is limited to two additional evidence passes and `10 minutes`.
* `GRD-ACT-005`: An unresolved conflict produces `continue_investigation`; critical incidents additionally transition to escalation.

State transitions:

```text
ASSESSMENT
→ SINGLE_ACTION_ELIGIBLE
→ CONFLICTING_ACTIONS
    → CONFLICT_RESOLUTION
        → RESOLVED
        → UNRESOLVED_CONTINUE_INVESTIGATION
```

Failure behavior:

* No `GuardianAction` resource may be created while the incident is in `CONFLICT_RESOLUTION`.
* Exhausting the conflict budget must not cause an arbitrary model choice.
* A conflicting action already executing causes the new proposal to be denied and the incident to gather fresh evidence.

### 4.7.4 Temporal ownership and expiry

Requirements:

* `GRD-WF-001`: The parent workflow ID is `guardian/{tenant_id}/incident/{incident_id}`.
* `GRD-WF-002`: `DeploymentAssessmentWorkflow`, `ActionExecutionWorkflow`, and `RecoveryVerificationWorkflow` must run as children of the incident parent.
* `GRD-WF-003`: Child workflow IDs must be deterministic and include the parent incident ID and immutable subject ID, such as assessment, proposal, or recovery-evaluation ID.
* `GRD-WF-004`: Parent cancellation or expiry must request child cancellation and wait for a safe checkpoint before closure.
* `GRD-TTL-001`: Default incident inactivity TTL is `24 hours` for `info` and `warning`, and `72 hours` for `critical`.
* `GRD-TTL-002`: An incident expiry extension requires an explicit audited operator or workflow transition; background retries must not silently extend expiry.
* `GRD-TTL-003`: Action proposal TTL defaults to `15 minutes` and must not exceed `30 minutes`.
* `GRD-TTL-004`: Approval nonce TTL is `5 minutes`.
* `GRD-TTL-005`: Approval validity is `10 minutes` or the remaining proposal TTL, whichever is shorter.
* `GRD-TTL-006`: OPA must be reevaluated no more than `30 seconds` before action execution starts.

State transitions:

```text
ACTIVE
→ EXPIRING
→ EXPIRED
    → CHILDREN_CANCELLED
    → PROPOSALS_EXPIRED
    → APPROVALS_REVOKED
    → CLOSED_WITH_EXPIRY
```

Failure behavior:

* Expired incidents, proposals, approvals, nonces, or policy decisions are non-executable.
* A delayed approval does not revive a proposal.
* A child that cannot reach a safe cancellation checkpoint causes escalation but does not permit another action to bypass the parent.

### 4.7.5 Identity evidence and hybrid topology

Requirements:

* `GRD-ID-001`: Version disagreement between authoritative observations must be persisted as immutable `identity_conflict` evidence.
* `GRD-ID-002`: Guardian must preserve every conflicting value and its provenance; it must not silently choose the first or highest-priority value.
* `GRD-ID-003`: Any unresolved version identity conflict blocks version-specific rollback.
* `GRD-ID-004`: Non-version-specific investigation and alerting may continue during identity resolution.
* `GRD-TOP-001`: Static service configuration is authoritative for intended ownership, service criticality, permitted action domains, and registry bindings.
* `GRD-TOP-002`: Runtime traces are authoritative for observed request dependency edges within the last `15 minutes`.
* `GRD-TOP-003`: Kubernetes owner references and controller state are authoritative for workload and controller ownership when the snapshot age is `<= 60 seconds`.
* `GRD-TOP-004`: SigNoz metadata is authoritative for telemetry attribute availability and signal identity when metadata age is `<= 5 minutes`.
* `GRD-TOP-005`: No topology source may overwrite fields owned by another source. Cross-source disagreement must be recorded as evidence.
* `GRD-TOP-006`: Action-domain calculation must use the latest non-conflicting snapshot from every required authority.

State transitions:

```text
IDENTITY_UNRESOLVED
→ IDENTITY_RESOLVED
→ IDENTITY_CONFLICT
    → IDENTITY_EVIDENCE_COLLECTED
    → IDENTITY_RESOLVED
    → VERSION_SPECIFIC_ACTION_BLOCKED
```

Failure behavior:

* Stale topology data blocks action-domain calculation.
* Missing static ownership may fall back to investigation but not to a narrower lock domain.
* A topology conflict cannot be resolved by LLM assertion.

### 4.7.6 Tenant isolation

Requirements:

* `GRD-TEN-001`: `tenant_id` is mandatory on every query, cache key, workflow ID, NATS subject, evidence path, provider connection, credential reference, policy decision, audit event, and database row.
* `GRD-TEN-002`: Ingress without an authenticated tenant is rejected; Guardian has no implicit default tenant.
* `GRD-TEN-003`: PostgreSQL must use row-level security or an equivalent database-enforced tenant predicate.
* `GRD-TEN-004`: Object storage prefixes must begin with the tenant ID.
* `GRD-TEN-005`: Shared caches must include tenant ID before policy, contract, or query identity.
* `GRD-TEN-006`: Temporal search attributes and workflow signals must carry tenant identity and reject mismatches.
* `GRD-TEN-007`: Provider credentials must be selected by tenant and provider connection ID; cross-tenant credential fallback is prohibited.
* `GRD-TEN-008`: Administrative cross-tenant access requires a dedicated role, explicit tenant selection, and an audit event.

Failure behavior:

* A missing or mismatched tenant fails closed before external I/O.
* Cache, NATS, Temporal, or database records with missing tenant scope are quarantined and never processed.
* Cross-tenant evidence references invalidate the assessment.

### 4.7.7 Query Contract GitOps and restricted builder AST

Requirements:

* `GRD-QC-001`: Git is the source of truth for Query Contract source definitions.
* `GRD-QC-002`: The lifecycle is `draft → static_validated → shadow → approval_pending → approved → superseded|retired`.
* `GRD-QC-003`: Static validation must check schema, tenant predicates, bounded ranges, cardinality, units, result shape, parameter allowlists, and backend compatibility.
* `GRD-QC-004`: Shadow validation requires at least `24 hours` and `100 executions`, `>= 99%` successful execution, zero result-shape violations, and zero cardinality-limit violations.
* `GRD-QC-005`: Approval requires two distinct approvers. Neither approver may be the author, and the two approvals must reference the same content hash.
* `GRD-QC-006`: An approved version is immutable.
* `GRD-QC-007`: Supersession must atomically set the new active version, mark the previous version superseded, and write an outbox event in one database transaction.
* `GRD-QC-008`: Runtime resolution must use an immutable version and content hash, never a mutable name alone.

Builder language requirements:

* `GRD-AST-001`: `language: builder` must contain a restricted typed AST, not an arbitrary string.
* `GRD-AST-002`: Allowed nodes are `Signal`, `Filter`, `Window`, `Rate`, `Aggregate`, `GroupBy`, `Arithmetic`, `Clamp`, and `ScalarResult`.
* `GRD-AST-003`: There is no raw-expression, SQL-fragment, PromQL-fragment, ClickHouse-fragment, template, function-call, or backend-escape node.
* `GRD-AST-004`: Maximum AST depth is `12`, maximum node count is `64`, maximum filters is `32`, and maximum group-by fields is `3`.
* `GRD-AST-005`: Every attribute, operator, aggregation, unit conversion, and group-by field must come from a versioned allowlist.
* `GRD-AST-006`: Compilation to a backend query occurs only in the SigNoz gateway and the compiled query is recorded in provenance.

Failure behavior:

* A static-analysis, shadow, approval, hash, or supersession failure leaves the previous approved version active.
* Partial supersession is rolled back transactionally.
* Unknown AST nodes or backend fragments are rejected before shadow execution.

### 4.7.8 Scaler consistency, policy propagation, and mTLS

Requirements:

* `GRD-SCL-001`: The SigNoz gateway result cache is the shared authoritative short-lived scaler result.
* `GRD-SCL-002`: Scaler replicas may use a local cache only as an optimization. Default local TTL is `2 seconds`.
* `GRD-SCL-003`: Default gateway cache TTL is `15 seconds` and must not exceed the Query Contract freshness limit.
* `GRD-SCL-004`: The cache key includes tenant ID, policy content hash, Query Contract ID and version, normalized parameters, and evaluation time bucket.
* `GRD-SCL-005`: A local value must include the gateway result ID and expiry and must never outlive the gateway result.
* `GRD-SCL-006`: Last-known-valid TTL is `min(2 × KEDA poll interval, 60 seconds, Query Contract freshness)`.
* `GRD-SCL-007`: A local/gateway disagreement is resolved in favor of the gateway and emits an integrity metric.
* `GRD-POL-001`: Signed policy and Query Contract updates are pushed over NATS.
* `GRD-POL-002`: Every replica polls the authoritative store every `60 seconds` to repair missed events.
* `GRD-POL-003`: Policy propagation P99 must be `<= 90 seconds`.
* `GRD-POL-004`: Each replica records desired version, applied version, signature status, bundle age, and last successful refresh.
* `GRD-TLS-001`: KEDA-to-scaler transport uses cert-manager-issued mutual TLS certificates.
* `GRD-TLS-002`: Certificate duration is `24 hours` with `renewBefore: 8h`.
* `GRD-TLS-003`: Client and server certificates use separate identities and extended-key usages, and SAN verification is mandatory.
* `GRD-TLS-004`: TLS 1.2 or newer is required; plaintext and server-only TLS listeners are prohibited.

Failure behavior:

* Gateway failure returns an error or unexpired last-known-valid value and must not permit scale-down.
* Missing, expired, untrusted, wrong-SAN, or wrong-EKU certificates fail closed.
* A missed NATS update must be repaired by polling; a missed push and failed poll enter the OPA staleness modes below.

### 4.7.9 OPA staleness tiers

Requirements:

* `GRD-OPA-001`: Bundle age `<= 5 minutes` is `FRESH`.
* `GRD-OPA-002`: Bundle age `> 5 minutes` and `<= 15 minutes` is `RESTRICTED`.
* `GRD-OPA-003`: Bundle age `> 15 minutes`, invalid signature, unknown desired version, or failed integrity verification is `FAIL_CLOSED`.
* `GRD-OPA-004`: `FRESH` permits normal policy evaluation.
* `GRD-OPA-005`: `RESTRICTED` permits read-only investigation, alert-only actions, and observation of already-running actions. It denies new rollback, scale-down, scaler pause, policy activation, approval issuance, and other destructive actions.
* `GRD-OPA-006`: `FAIL_CLOSED` denies all new writes, including scale-up, and the scaler must return an error or safe hold behavior.
* `GRD-OPA-007`: An already-running action may finish its current atomic provider operation but must not begin another mutation step after entering `FAIL_CLOSED`.

State transitions:

```text
FRESH
→ RESTRICTED
→ FAIL_CLOSED
→ FRESH
```

Failure behavior:

* Read-only incident investigation remains available in every tier.
* Bundle recovery requires successful signature verification and an applied version equal to the desired version.
* Clock uncertainty greater than `60 seconds` forces `FAIL_CLOSED`.

### 4.7.10 Dependency-aware action concurrency and operator drift

Requirements:

* `GRD-CON-001`: Guardian locks action domains, not only direct workloads.
* `GRD-CON-002`: An action domain includes the target workload, owning controller, related HPA/KEDA/ScaledObject, traffic-routing object, rollback provider object, and dependencies explicitly marked `actionCoupled`.
* `GRD-CON-003`: All lock keys are calculated before execution, sorted, and acquired atomically.
* `GRD-CON-004`: Lock lease duration is `60 seconds` with heartbeat every `20 seconds`.
* `GRD-CON-005`: Any overlapping action-domain key blocks the new write action.
* `GRD-DRIFT-001`: Before every mutation step, Guardian compares protected resource fields, generation, resource version, and provider revision with the approved execution preconditions.
* `GRD-DRIFT-002`: Changes carrying the current Guardian execution identity are expected drift; other protected-field changes are manual or external drift.
* `GRD-DRIFT-003`: Unexpected drift transitions the action to `SUPERSEDED_BY_OPERATOR`, releases locks, and prevents further writes.
* `GRD-DRIFT-004`: Guardian must not restore its prior desired state over an operator change.
* `GRD-DRIFT-005`: Resuming automation after drift requires fresh evidence, a new proposal, and a new policy decision.

State transitions:

```text
READY
→ DOMAIN_LOCK_PENDING
→ DOMAIN_LOCKED
→ EXECUTING
    → COMPLETED
    → FAILED
    → SUPERSEDED_BY_OPERATOR
    → LOCK_LOST
```

Failure behavior:

* Partial domain-lock acquisition is rolled back.
* Lock loss stops new mutation steps and escalates.
* Metadata-only changes outside the protected-field allowlist do not supersede an action.

### 4.7.11 Approved hierarchical Recovery Contract registry

Requirements:

* `GRD-REC-001`: Recovery Contracts are resolved from an approved hierarchy:
  `global → tenant → environment → service_class → service → workload_and_action`.
* `GRD-REC-002`: The most specific compatible approved contract wins.
* `GRD-REC-003`: Contract versions are immutable and carry tenant, scope, parent, approvers, content hash, and registry version.
* `GRD-REC-004`: Every mutating action proposal pins the resolved Recovery Contract version and registry version before policy evaluation.
* `GRD-REC-005`: The LLM may recommend a registry lookup but cannot create or modify a runtime Recovery Contract.
* `GRD-REC-006`: Missing, ambiguous, retired, or incompatible recovery resolution blocks the action.
* `GRD-REC-007`: A child contract may tighten inherited conditions. Removing or weakening an inherited condition requires explicit override approval.

Failure behavior:

* Registry resolution failure produces continue-investigation or alert-only behavior.
* A registry update does not change an already approved proposal.
* Recovery is evaluated only against the pinned contract version.

### 4.7.12 Model, prompt, replay, and air-gapped inference lifecycle

Requirements:

* `GRD-MLO-001`: Model and prompt changes progress through deterministic replay, statistical model-in-the-loop replay, live shadowing, canary, and production.
* `GRD-MLO-002`: Deterministic scaffold replay performs no live model call and must reproduce deterministic features, eligibility, state transitions, and action gates byte-for-byte.
* `GRD-RPL-001`: Statistical replay runs the candidate model at least `20` times per critical scenario and reports distributions for classification, tie-break selection, unsupported claims, abstention, tool use, and latency.
* `GRD-MLO-003`: Offline evaluation must include at least `200` incidents and every critical safety scenario.
* `GRD-MLO-004`: Live shadowing lasts at least `7 days` and covers at least `100` incidents.
* `GRD-MLO-005`: Canary promotion uses `10%` of eligible tenants for at least `24 hours`, then `50%` for at least `24 hours`, before full promotion.
* `GRD-MLO-006`: Promotion requires zero increase in dangerous-action rate, no more than `2 percentage points` regression in correct abstention, invalid structured output rate `< 0.5%`, and model-call P95 within the configured SLO.
* `GRD-MLO-007`: Model and prompt activation uses one atomic version pointer. Rollback changes the pointer atomically and must propagate within `90 seconds`.
* `GRD-INF-001`: The initial air-gapped profile supports models from `14B` through `32B` parameters.
* `GRD-INF-002`: Production starts with two inference replicas.
* `GRD-INF-003`: Global model-involved investigation concurrency is configurable from `8` to `16`; default is `12`.
* `GRD-INF-004`: Default per-tenant model-involved concurrency is `4`.
* `GRD-INF-005`: Model queue wait is capped at `30 seconds` and model-call duration at `60 seconds`.
* `GRD-INF-006`: When inference is unavailable, deterministic non-tie assessments may proceed; near-ties transition to continued investigation.
* `GRD-INF-007`: Inference replicas and model gateways have no public egress in the air-gapped profile.

Failure behavior:

* Any dangerous-action regression automatically blocks or rolls back promotion.
* A failed atomic activation leaves the prior model and prompt pair active.
* Queue or timeout exhaustion must not create a model-derived winner.

### 4.7.13 Transactional outbox

Requirements:

* `GRD-OUT-001`: Inbound event acceptance, incident state changes, proposal creation, approval recording, Query Contract supersession, and policy activation must write an outbox record in the same PostgreSQL transaction as the domain change.
* `GRD-OUT-002`: API and domain services must not directly start Temporal workflows or publish authoritative NATS events before the transaction commits.
* `GRD-OUT-003`: The outbox dispatcher publishes with a stable idempotency key and records acknowledgement.
* `GRD-OUT-004`: Dispatch retry uses exponential backoff from `1 second` to `60 seconds`.
* `GRD-OUT-005`: Outbox lag warning threshold is `60 seconds`; paging threshold is `5 minutes`.
* `GRD-OUT-006`: After `20` failed attempts, the record is quarantined for operator review while the related automation remains stalled and non-executable.
* `GRD-OUT-007`: There is no bypass path that directly executes an action because NATS or Temporal is unavailable.

State transitions:

```text
PENDING
→ PUBLISHING
→ ACKNOWLEDGED
→ RETRY_WAIT
→ QUARANTINED
```

Failure behavior:

* PostgreSQL commit without NATS or Temporal availability is safe: the state and pending outbox record remain durable, and automation stalls.
* NATS acknowledgement without local acknowledgement is retried idempotently.
* Quarantine creates an alert and audit event but never a direct-execution fallback.

# 9. Incident workflow

## 9.1 Workflow states

Each incident is controlled by exactly one Temporal `IncidentWorkflow` parent.

```text
RECEIVED
→ NORMALIZED
→ CORRELATED
→ EVIDENCE_COLLECTION
→ TELEMETRY_VALIDATION
    → TELEMETRY_FAILURE
    → ASSESSMENT
        → UNKNOWN
        → CONFLICT_RESOLUTION
            → EVIDENCE_COLLECTION
            → CONTINUE_INVESTIGATION
            → ASSESSMENT
        → ACTION_PROPOSED
→ POLICY_EVALUATION
    → DENIED
    → APPROVAL_REQUIRED
        → APPROVAL_EXPIRED
        → APPROVED
    → AUTOMATICALLY_AUTHORIZED
→ ACTION_DOMAIN_LOCK
    → BLOCKED_BY_CONCURRENCY
    → ACTION_EXECUTION_CHILD
        → SUPERSEDED_BY_OPERATOR
        → ACTION_FAILED
        → ACTION_COMPLETED
→ RECOVERY_WAIT
→ RECOVERY_EVALUATION_CHILD
    → RECOVERED
    → NOT_RECOVERED
    → TELEMETRY_UNHEALTHY
→ CLOSED or ESCALATED

Any active state
→ EXPIRING
→ EXPIRED
→ CLOSED_WITH_EXPIRY
```

The parent starts `DeploymentAssessmentWorkflow`, `ActionExecutionWorkflow`, and `RecoveryVerificationWorkflow` as deterministic child workflows. The parent owns timers, approvals, cancellation, expiry, conflict resolution, and the final incident state.

## 9.2 Incident correlation

Recommended fingerprint:

```text
environment
+ service
+ alert rule
+ normalized failing operation
+ dependency
+ rolling time bucket
```

Rules:

* Related alerts join an active incident within the same tenant.
* Late deployment events may signal the existing incident parent.
* Dependency incidents may be linked to multiple caller incidents without merging their parent workflows.
* A duplicate start request for the same `tenant_id` and `incident_id` must signal or return the existing parent.
* A resolved SigNoz alert does not automatically close the Guardian incident.
* Guardian closes only after Recovery Contract success, explicit operator closure, cancellation, or deterministic expiry.
* Expired incidents cannot be reopened; a new source event creates or correlates to a new incident.

## 10.4 Evidence scoring

```text
evidence_score =
    relevance
  × source_quality
  × freshness
  × sample_adequacy
  × identity_confidence
```

```text
hypothesis_score =
  sigmoid(
      independent_support
    - independent_contradiction
    - missing_evidence_penalty
  )
```

Normative gates:

```text
hypothesis eligible:
  deterministic_score >= 0.70
  AND normalized_contradiction_weight <= 0.25
  AND telemetry_quality >= 0.80
  AND no critical telemetry or identity failure

mutating action eligible:
  hypothesis eligible
  AND evidence_confidence >= 0.85

near tie:
  every tied hypothesis eligible
  AND max(score) - min(score) <= 0.03
```

Evidence from the same underlying dataset must not be counted as multiple independent confirmations. The deterministic scorer assigns eligibility and rank before the model runs. The model may provide only `model_tie_break_rank` for a near-tie group and may not alter any deterministic field.

A simultaneously eligible set that recommends incompatible actions enters `CONFLICT_RESOLUTION`. The default conflict budget is two additional evidence passes and `10 minutes`; unresolved conflicts produce `continue_investigation`.

## 10.5 Telemetry-quality checks

Guardian must check:

* Data freshness.
* Service-version attributes.
* Trace sampling.
* Log ingestion delay.
* Cardinality collapse.
* Collector restarts.
* Sampling configuration changes.
* Timestamp skew.
* Environment identity.
* Workload identity.
* Tenant identity.
* Collector and ingestion-pipeline availability.

Classification boundary:

```text
telemetry_failure:
  telemetry_quality < 0.80
  OR any critical telemetry integrity check fails

unknown:
  telemetry_quality >= 0.80
  AND no critical telemetry integrity check fails
  AND no causal hypothesis remains eligible after the evidence budget
```

Critical integrity failures include identity conflicts, required samples older than twice the Query Contract freshness limit, timestamp skew greater than `60 seconds`, zero usable samples for a required signal, known collector outage, and comparison-invalidating cardinality or sampling changes.

Guardian must never classify unhealthy telemetry as `unknown`. A telemetry-quality evaluator error produces `telemetry_failure`. Mutating actions are prohibited while the incident class is `telemetry_failure`.

## 10.6 Hybrid topology authority

Guardian builds a versioned topology snapshot from four authorities:

| Source | Authoritative fields | Default freshness |
| --- | --- | ---: |
| Static service configuration | Intended ownership, criticality, allowed action domains, registry bindings | Current approved version |
| Runtime traces | Observed request and dependency edges | 15 minutes |
| Kubernetes state | Owner references, controllers, workloads, rollout identity | 60 seconds |
| SigNoz metadata | Telemetry attributes, signal identity and availability | 5 minutes |

No source may overwrite fields owned by another source. Disagreements are immutable evidence. Missing or stale required topology blocks action-domain calculation and mutating actions; read-only investigation continues.

## 12.6 Failure behavior

On SigNoz query failure:

* Never return manufactured zero.
* Never allow scale-down.
* Return an error or approved last-known-valid value.
* Limit last-known-valid data to `min(2 × KEDA poll interval, 60 seconds, Query Contract freshness)`.
* Alert after repeated failures.
* Preserve current replicas.

On consistency or policy failure:

* Gateway result wins over a conflicting local value.
* A local value without a valid gateway result ID or unexpired gateway expiry is discarded.
* `RESTRICTED` OPA mode denies destructive scaler behavior and scale-down.
* `FAIL_CLOSED` OPA mode denies all new scaler writes and returns error or safe hold behavior.
* Missing tenant scope, policy hash mismatch, Query Contract mismatch, or untrusted mTLS identity fails closed.

The original safety requirements explicitly prohibit manufacturing zero and require no scale-down when the query is unhealthy.

## 14.3 Bundle propagation and staleness

Signed bundle updates are pushed over tenant-scoped NATS subjects and repaired by polling the authoritative store every `60 seconds`.

```text
FRESH:
  bundle age <= 5m
  → normal policy evaluation

RESTRICTED:
  5m < bundle age <= 15m
  → read-only investigation
  → alert-only actions
  → observe already-running actions
  → deny rollback, scale-down, scaler pause, policy activation and new approvals

FAIL_CLOSED:
  bundle age > 15m
  OR invalid signature
  OR unknown desired version
  OR clock uncertainty > 60s
  → deny all new writes, including scale-up
  → scaler returns error or safe hold
  → read-only investigation remains available
```

A replica returns to `FRESH` only after signature verification succeeds and its applied version equals the desired version. Policy propagation P99 must be `<= 90 seconds`.

* Produce a new assessment.

## 15.4 Recovery Contract registry

```text
global
→ tenant
→ environment
→ service_class
→ service
→ workload_and_action
```

* A child may tighten inherited conditions.
* Removing or weakening an inherited condition requires explicit override approval.
* The LLM cannot create, edit, or substitute a runtime Recovery Contract.
* A later registry update does not alter an already approved proposal.

## 24.2 Property-based invariants

* Stale metrics never authorize scale-down.
* Duplicate events create at most one incident parent workflow.
* Duplicate actions execute at most once.
* Removing evidence cannot increase confidence.
* Contradicting evidence cannot improve a hypothesis.
* An ineligible hypothesis cannot become eligible through model output.
* A model cannot change deterministic score or rank outside a near-tie group.
* Unhealthy telemetry cannot produce `unknown`.
* Conflicting mutating actions cannot create a `GuardianAction`.
* Expired incidents, proposals, approvals, nonces, and policy decisions cannot execute.
* Policy hash mismatch cannot return a metric.
* A local scaler cache entry cannot outlive its gateway result.
* Recovery cannot succeed with unhealthy telemetry.
* Cross-tenant data cannot affect another tenant's assessment, cache, workflow, action, or recovery.
* Partial action-domain lock acquisition cannot persist.
* Unexpected protected-field drift cannot be overwritten by Guardian.
* Replica recommendations remain in bounds.
* Rollback target cannot equal the current version.
* Unresolved identity conflict cannot authorize version-specific rollback.
* Query Contract supersession exposes either the old active version or the new active version, never a partial state.

## 24.5 Incident replay suite

Required scenarios:

```text
legitimate load spike
bad deployment
dependency outage

```

A replay includes:

* Tenant and source events.
* Telemetry snapshots and quality state.
* Deployment changes.
* Kubernetes state.
* Hybrid topology snapshot.
* Code diff.
* Deterministic scoring version and expected scores.
* Expected hypotheses and eligibility.
* Rejected hypotheses.
* Expected conflict-resolution transitions.
* Allowed and forbidden actions.
* Recovery Contract and registry version.
* Policy bundle version and staleness state.
* Recovery expectation.

Replay has two required modes:

1. **Deterministic scaffold replay.** No live model call is made. Deterministic features, scores, eligibility, state transitions, action gates, and recovery evaluation must reproduce byte-for-byte.
2. **Statistical model-in-the-loop replay.** Each critical scenario runs at least `20` times and reports distributions for classification, near-tie selection, unsupported claims, abstention, tool use, latency, and dangerous-action rate.

## 24.6 Model and prompt evaluation

Track:

* Classification accuracy.
* Unsupported causal claims.
* Evidence citation precision.
* Contradiction detection.
* Correct abstention.
* Correct action set.
* Dangerous-action rate.
* Tool usage.
* Latency.
* Repeatability.
* Invalid structured output.
* Near-tie override attempts.
* Per-tenant and global concurrency behavior.

Promotion sequence:

```text
deterministic replay
→ statistical model-in-the-loop replay
→ live shadow for >= 7 days and >= 100 incidents
→ 10% tenant canary for >= 24h
→ 50% tenant canary for >= 24h
→ production
```

Offline evaluation must include at least `200` incidents and every critical safety scenario. Promotion requires zero increase in dangerous-action rate, no more than `2 percentage points` regression in correct abstention, invalid structured output below `0.5%`, and model-call P95 within SLO.

Model and prompt activation uses one atomic version pointer. Any dangerous-action regression blocks or automatically rolls back promotion; rollback must propagate within `90 seconds`.

## 24.7 Normative architecture acceptance tests

| ID | Given | When | Then |
| --- | --- | --- | --- |
| `AT-REA-001` | An ineligible hypothesis is model-ranked first | Assessment is finalized | It remains ineligible and cannot produce a mutating proposal |
| `AT-REA-002` | Two eligible hypotheses differ by `0.03` or less | The model tie-break runs | Only `model_tie_break_rank` may change; deterministic fields remain byte-identical |
| `AT-REA-003` | The top two eligible hypotheses differ by more than `0.03` | The model prefers the second | The deterministic winner remains authoritative |
| `AT-CLS-001` | Telemetry quality is `0.79` or a critical integrity check fails | Classification runs | Incident class is `telemetry_failure`, never `unknown` |
| `AT-CLS-002` | Telemetry quality is at least `0.80`, all integrity checks pass, and no hypothesis is eligible after the budget | Classification runs | Incident class is `unknown` |
| `AT-ACT-001` | Eligible hypotheses recommend scale and rollback | Assessment completes | Incident enters `CONFLICT_RESOLUTION` and no `GuardianAction` is created |
| `AT-WF-001` | Duplicate events arrive concurrently | Correlation starts workflows | Exactly one parent workflow exists for the tenant and incident |
| `AT-TTL-001` | A proposal or approval expires before execution | Execution is attempted | OPA and the controller deny it; no provider mutation occurs |
| `AT-ID-001` | `service.version`, image digest mapping, and deployment event disagree | Rollback assessment runs | `identity_conflict` evidence is stored and version-specific rollback is blocked |

| `AT-TEN-001` | Tenant A references Tenant B evidence, cache, workflow signal, or credential | The request is processed | It is rejected before external I/O and an audit/security event is emitted |
| `AT-QC-001` | A Query Contract has one approval or insufficient shadow evidence | Activation is attempted | The previous approved version remains active |
| `AT-QC-002` | Supersession fails between active-pointer and status updates | The transaction aborts | Readers observe only the previous active version |
| `AT-AST-001` | A builder AST contains a raw fragment, unknown node, excessive depth, nodes, filters, or grouping | Static analysis runs | The contract is rejected before shadow execution |
| `AT-SCL-001` | Two scaler replicas hold different local values | Both contact the gateway | Both converge on the gateway result ID; stale local values are discarded |
| `AT-SCL-002` | SigNoz and the gateway are unavailable beyond last-known-valid TTL | KEDA polls | The scaler returns error/hold and never enables scale-down |
| `AT-POL-001` | A replica misses a NATS policy event | The `60-second` poll runs | It applies the desired signed version within `90 seconds` P99 |
| `AT-OPA-001` | Bundle age crosses `5 minutes` and then `15 minutes` | Policy-dependent requests arrive | The replica transitions `FRESH → RESTRICTED → FAIL_CLOSED` with the specified write denials while read-only investigation remains |
| `AT-TLS-001` | The client certificate is expired, untrusted, wrong-SAN, wrong-EKU, or plaintext | KEDA calls the scaler | The connection fails closed |

| `AT-MLO-001` | A candidate model increases dangerous-action rate above baseline | Promotion or canary evaluation runs | Promotion is blocked or the atomic pointer rolls back within `90 seconds` |
| `AT-RPL-001` | A deterministic replay is run repeatedly | Results are compared | Scores, eligibility, transitions, and gates are byte-identical |
| `AT-RPL-002` | A critical scenario is run in model-in-the-loop mode | Evaluation completes | At least `20` runs and distributional metrics are recorded |
| `AT-OUT-001` | PostgreSQL commits while NATS or Temporal is unavailable | Dispatch fails | The outbox retries, automation remains stalled, and no bypass action executes |
| `AT-INF-001` | One of two inference replicas is unavailable | Eight investigations run concurrently | The system remains within latency/error limits and near-ties fail safely |

# 29. Production acceptance criteria

## SRE Sidekick

* Always returns a valid typed assessment.
* Distinguishes all six incident classes.
* Deterministic scoring assigns eligibility and authoritative rank before model execution.
* The model changes only the tie-break rank for eligible hypotheses within a `0.03` score window.
* `telemetry_quality < 0.80` or a critical integrity failure always produces `telemetry_failure`.
* Healthy telemetry with no eligible causal hypothesis after the evidence budget produces `unknown`.
* Records supporting, contradicting, identity-conflict, topology, and telemetry-quality evidence.
* Conflicting mutating recommendations enter conflict resolution and never create an action directly.
* Cannot call write tools.
* Links every explanation to evidence.
* Passes critical scenarios in multiple environments.

## Deploy Guardian

* Ingests real tenant-scoped deployment events through the transactional outbox.
* Maps deployments to observed versions.
* Persists version disagreement as `identity_conflict` evidence.
* Blocks version-specific rollback while identity is unresolved.
* Compares old and new versions.
* Retrieves relevant changes.
* Requires approval for production rollback.
* Rejects expired incidents, proposals, approvals, nonces, and policy decisions.
* Executes idempotently with dependency-aware action-domain locks.
* Yields to manual operator drift and records `SUPERSEDED_BY_OPERATOR`.
* Pins an approved Recovery Contract from the hierarchical registry.
* Verifies recovery with fresh telemetry.
* Works through at least two rollback providers.

## Guardian Scale

* Produces reviewable policies.
* Requires shadow mode before activation.
* Uses only approved, immutable, two-person-approved Query Contract versions.
* Rejects arbitrary builder fragments and compiles only the restricted typed AST.
* Keeps the LLM outside the polling path.
* Uses the SigNoz gateway as the shared authoritative short-lived result and local caches only as a `2-second` optimization.
* Never manufactures zero.
* Never scales down on stale telemetry.
* Enforces OPA `FRESH`, `RESTRICTED`, and `FAIL_CLOSED` behavior.
* Uses cert-manager-issued mTLS certificates and rejects invalid identities.
* Enforces bounds and blockers.
* Exposes control-loop telemetry.
* Passes KEDA conformance and multi-replica consistency tests.

## Query Contracts and policy

* Git is the source of truth.
* Static analysis, at least `24 hours` and `100 executions` of shadow validation, and two-person approval are enforced.
* Approved versions are immutable.
* Supersession is atomic and leaves the prior version active on failure.
* NATS pushes updates and `60-second` polling repairs missed events.
* Policy propagation P99 is `<= 90 seconds`.
* Read-only investigation remains available during policy staleness.
* Destructive actions stop in `RESTRICTED`; all new writes stop in `FAIL_CLOSED`.

## Workflow, tenancy, and durability

* Exactly one Temporal parent exists per tenant and incident.
* Deployment assessment, action execution, and recovery verification execute as child workflows.
* Incident and approval expiry timers are durable and deterministic.
* Every query, cache key, workflow ID, NATS subject, evidence path, provider credential, policy decision, audit event, and database row is tenant-scoped.
* Cross-tenant access tests fail before external I/O.
* PostgreSQL and the transactional outbox remain authoritative when NATS or Temporal is unavailable.
* NATS or Temporal failure stalls automation safely without a bypass path.

## Recovery, model lifecycle, and replay

* Recovery Contracts resolve deterministically from the approved hierarchy and are pinned before action authorization.
* Deterministic scaffold replay reproduces scores, eligibility, transitions, and gates byte-for-byte.
* Statistical model-in-the-loop replay runs each critical scenario at least `20` times.
* Models and prompts pass replay, at least `7 days` and `100 incidents` of live shadowing, `10%` canary, and `50%` canary before full promotion.
* Any dangerous-action regression blocks or atomically rolls back promotion within `90 seconds`.
* The air-gapped profile runs two `14B–32B` inference replicas and supports `8–16` concurrent investigations.

# 30. Key risks

| Risk | Mitigation |
| --- | --- |
| Overfitting to OpenTelemetry Demo | Multi-environment validation |
| SigNoz API changes | Versioned gateway adapter |
| Weak or conflicting service-version data | Immutable identity-conflict evidence and rollback block |
| Unsupported model claims | Evidence linking, deterministic authority, and near-tie-only model role |
| Prompt injection | Untrusted evidence handling and no write tools |
| Conflicting scale and rollback recommendations | Conflict-resolution state and no mutating default |
| Duplicate action | Idempotency and one Temporal parent |
| Concurrent actions on related resources | Dependency-aware atomic action-domain locks |
| Guardian fights an operator | Protected-field drift detection and `SUPERSEDED_BY_OPERATOR` |
| Stale scaling data | Gateway-authoritative cache, strict last-known-valid TTL, and no scale-down |
| Dependency overload | Blocking Query Contracts |
| Query injection or explosion | Restricted typed AST, static analysis, cardinality and cost limits |
| Query Contract partial promotion | Transactional atomic supersession |
| GitOps drift | Provider-aware rollback and immutable contract versions |
| Approval spoofing or reuse | Signed nonce, OIDC identity, TTL, and immediate OPA reevaluation |
| Cross-tenant access | Mandatory tenant scope, database enforcement, and negative isolation tests |
| Workflow loss | Temporal parent and child workflows |
| NATS or Temporal outage after commit | Transactional outbox and safe automation stall |
| Event loss | JetStream persistence and idempotent outbox delivery |
| Policy mismatch or staleness | Signed bundles, push plus poll, and tiered fail-closed modes |
| mTLS expiry or identity error | cert-manager rotation, SAN/EKU verification, and fail-closed transport |
| Recovery criteria inconsistency | Approved hierarchical Recovery Contract registry |
| Model or prompt regression | Replay, shadow, canary, atomic activation and rollback |
| Air-gapped inference overload | Two replicas, bounded `8–16` concurrency, queue and call limits |
| Automation activated early | Mandatory shadow mode |
| Demo repository changes | Pin commit SHAs |

Guardian is not production-ready merely when it can identify an incident or execute an action.

It is production-ready only when it can:

* Explain why an action is appropriate.
* Show the evidence supporting it.
* Identify contradicting evidence.
* Obtain the required policy decision.
* Execute idempotently.
* Verify recovery.
* Preserve a complete audit trail.
* Demonstrate the same behavior across unrelated Kubernetes application environments.
